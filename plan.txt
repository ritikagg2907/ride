SECTION 0 — HIGH-LEVEL ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Services:
  1. api-gateway          NGINX reverse proxy — rate limit, JWT passthrough, routing
  2. user-service         Rider auth (register, OTP login, JWT, profile)
  3. driver-service       Driver auth, GPS location (Redis GEO), status
  4. trip-service         Lifecycle: REQUESTED→STARTED→COMPLETED, fare, ratings, WebSocket
  5. matching-service     Kafka consumer — geo-match, Redlock, reassign on decline
  6. surge-pricing-svc    Dedicated surge: H3 geo-cells, supply/demand → Redis cache
  7. payment-service      PSP (Razorpay/cash/UPI), circuit breaker, reconciliation
  8. notification-svc     Kafka fan-out → email (Brevo/SMTP) + in-app
  9. admin-service        Feature flags, kill-switches, DLQ replay, health rollup

Data Stores:
  Redis    — driver geo-index (GEORADIUS), surge cache, OTP, idempotency keys,
             session tokens, Redlock (dispatch), notification dedup
  Kafka    — all async event flows (KRaft mode, no Zookeeper)
  Postgres — one DB per service (per-service isolation)

Kafka Topics (UPPER_SNAKE_CASE):
  RIDE_REQUESTED          rider request published by trip-service
  RIDE_OFFERED            driver offer published by matching-service
  RIDE_CANCELLED          cancellation by rider/driver
  DRIVER_ASSIGNED         matching confirmed, consumed by trip + notification
  TRIP_STARTED            trip lifecycle event
  TRIP_COMPLETED          triggers payment + notification
  TRIP_CANCELLED          triggers refund + notification
  RATING_SUBMITTED        triggers user/driver score update
  PAYMENT_COMPLETED       triggers notification + driver earnings record
  PAYMENT_FAILED          triggers retry logic + rider notification
  SURGE_RECOMPUTE         periodic/supply-change trigger for surge-svc
  ADMIN_FLAG_CHANGED      feature flag broadcast to all services
  *.DLQ                   dead-letter suffix for every consumer group

SECTION 5 — SERVICE: MATCHING-SERVICE  (port 8087, background only)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

No HTTP — pure Kafka consumer + Redis.

[ ] 5.1  Kafka consumer on RIDE_REQUESTED:
         a. GEORADIUS drivers:geo → up to 20 nearest available drivers (5km radius)
            filter by vehicle_type match
         b. Score candidates: score = 0.6*(1/dist_km) + 0.3*rating + 0.1*acceptance_rate
         c. Take top-5; acquire Redlock per driver (TTL 15s, prevent double-assign)
         d. Publish RIDE_OFFERED to that driver; Redis key offer:{ride}:{driver}=pending TTL=30s
         e. Wait for driver response (poll Redis key); timeout = 30s
         f. On ACCEPT → call trip-service PATCH /trips/{id}/assign (internal),
            publish DRIVER_ASSIGNED, release locks on other candidates
         g. On DECLINE/timeout → release lock, try next candidate
         h. After 10 min of retries or radius expanded to 10km still no match →
            PATCH trip status=CANCELLED, publish RIDE_CANCELLED
[ ] 5.2  Redis pub/sub channel driver:response:{ride_id} for real-time driver ack
[ ] 5.3  Prometheus: dispatch_latency_seconds histogram (p95 target ≤ 1s),
         dispatch_success_total, dispatch_no_driver_total
[ ] 5.4  GET /health/ready (minimal HTTP server just for health check)

SECTION 6 — SERVICE: SURGE-PRICING-SERVICE  (port 8084)  [NEW — not in ref repo]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose: Dedicated surge micro-service. Trip-service reads from Redis only.
Uses H3 hex-cell indexing (resolution 7, ~5km²) for geo-cell granularity.

Redis keys:
  surge:cell:{h3_index}       multiplier FLOAT, TTL 90s
  surge:demand:{h3_index}     INCR counter, TTL 60s (sliding window)
  surge:override:{h3_index}   admin override, TTL variable

[ ] 6.1  Kafka consumer RIDE_REQUESTED → INCR surge:demand:{cell} (60s TTL)
[ ] 6.2  Kafka consumer SURGE_RECOMPUTE → for each cell in payload:
         supply  = GEORADIUS count of available drivers in cell
         demand  = GET surge:demand:{cell}
         ratio   = demand / max(supply, 1)
         mult    = clamp(1.0 + (ratio-1)*0.5, 1.0, 5.0)
         check   surge:override:{cell} → use override if present
         SET surge:cell:{cell} mult EX 90
[ ] 6.3  Background ticker every 30s → publish SURGE_RECOMPUTE for all active cells
[ ] 6.4  GET /surge/{h3_cell}                — read from Redis, fallback 1.0
[ ] 6.5  POST /surge/{h3_cell}/override      — admin: set override multiplier + TTL
         (requires X-Admin-Token header)
[ ] 6.6  GET /surge/map                      — return all active cell multipliers
         (reads all keys surge:cell:* from Redis SCAN)
[ ] 6.7  Prometheus: surge_multiplier gauge (per cell), recompute_duration histogram
[ ] 6.8  GET /health/ready

[ ] 11.1 Idempotency: X-Idempotency-Key on all mutation APIs; Redis SetNX stores
         key→response (TTL 24h); DB UNIQUE constraint as second guard

[ ] 11.2 Retries + jitter: PSP 2^n*1s ±20%, max 5 attempts, max delay 60s;
         Kafka producer retries=3 with 100ms base; consumer at-least-once

[ ] 11.3 Circuit breakers (gobreaker on PSP): open after 5 failures, half-open 30s;
         surge fallback: breaker OPEN → return multiplier=1.0 from Redis

[ ] 11.4 DLQ: every consumer on message failure after maxRetries publishes to
         {TOPIC}.DLQ; admin-service replay endpoint for recovery

[ ] 11.5 Backpressure: driver location updates: bounded async buffer (10k),
         drop+metric on full; matching: semaphore 500 goroutines, else 503

[ ] 11.6 Graceful shutdown: SIGTERM → drain 30s → flush Kafka → close pgxpool → exit

[ ] 11.7 Health probes on every service:
         GET /health/live  → 200 always (process alive)
         GET /health/ready → 200 if DB+Redis+Kafka OK, else 503
